## 프롬프트

`car_test/docs/` 아래 문서들의 스타일을 따라 새 기술 문서의 **뼈대(outline)만** 잡아줘.
내용을 대신 쓰지 말고, 각 섹션에 내가 뭘 채워야 하는지 한 줄 힌트만 남겨줘.

### 이 프로젝트 문서의 말투·구조 규칙

**형식**
- 언어: 한국어. 기술 용어(MPPI, ray-wheel, damping, penetration depth 등)는 영어 그대로.
- 파일명: `[YY-MM-DD]_주제.md` (핵심 개념 문서는 루트, 트러블슈팅·설정 등 심화 노트는 `tech/`).
- 최상단 `# 제목`은 "무엇을 했는가"가 드러나게 (예: "Ray+BVH를 활용한 충돌로의 전환").
- 제목 바로 아래 `>` 인용구로 **한 줄 요약** 또는 핵심 정의를 먼저 던진다.

**구조**
- 도입부에서 **문제/한계 → 왜 이 접근을 택했나 → 이 문서가 다루는 범위**를 2~4문장으로 압축.
- 본문은 번호형 접근(`시도 방법1`, `시도 방법2`)이나 단계형(`Stage 1`, `알고리즘 4단계`)으로.
- 실패한 시도도 숨기지 말고 `문제점` / `잘못된 이해` 같은 소제목으로 남긴다 (이게 이 문서들의 핵심 성격).
- 파라미터·비교·결과는 **표**로. 수식은 `$$...$$`, 코드/의사코드·수치 블록은 ```` ``` ````.
- 이미지·영상은 `![](../res_wjdaksry/날짜/파일)` 상대경로.

**말투**
- 짧고 단정적인 문장. "~했다 / ~이다 / ~한다" 종결. 격식체 아님.
- 인과를 화살표로 압축하는 습관: `가속 → pitch → vel[2] 누적 → 발산`.
- `> ` 인용구로 핵심 통찰·주의·직관을 툭 던진다 (예: "실제 차량도 rebound 댐핑이 bump보다 강함").
- **핵심 아이디어**, **잘못된 이해**, **문제점** 처럼 굵은 라벨로 요점을 표시.
- 과장 금지: 근거 없는 수치·완료 주장 금지. 검증된 것과 계획된 것을 분리해 쓴다.

### 출력 형식
- 위 규칙에 맞춘 **마크다운 뼈대만** 출력: 제목, 한 줄 요약 자리, 섹션 헤더, 표/코드블록 자리(빈 틀).
- 각 섹션 아래에 `<!-- 여기: ○○ 채우기 -->` 형태의 힌트 주석 한 줄.
- 실제 설명 문장·수치는 쓰지 마. 내가 내 언어로 채운다.
