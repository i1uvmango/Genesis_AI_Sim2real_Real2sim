## time vs pos : fair metric 설정

좋은 문제 제기고, 진단부터 말하면: 현재 표는 4개 지표 중 3개(fm, lag, 그리고 HE도 시간 인덱스 기준이라면 3.5개)가 "시간 결합" 지표라서, 시합 자체가 time mode의 홈그라운드에서 열린 겁니다. 목표가 경로 추종(공간)인데 채점은 궤적 추적(시공간)으로 한 것 — 불공정의 정체가 이겁니다.
1. 문헌적 근거 — 이 구분은 제어이론에서 정식화된 문제입니다

Path following vs trajectory tracking: Aguiar, Hespanha, Kokotović, "Performance limitations in reference tracking and path following for nonlinear systems" (Automatica, 2008)이 이 둘을 형식적으로 구분하고, trajectory tracking(시간 인덱스 목표)에는 근본적 성능 한계가 있지만 path following(공간 목표 + 자유로운 시간 매개화)은 그 한계를 회피할 수 있음을 증명했습니다. 사용자 목표("속도 오차가 있더라도 경로를 잘 따라가는 것")는 문자 그대로 path following의 정의이고, 따라서 fm/lag 같은 시간 결합 오차로 채점하는 건 문제 정의와 지표의 불일치입니다. Paden et al.의 자율주행 서베이(IEEE T-IV, 2016)도 같은 구분을 표준 용어로 씁니다.
매개화-불변 경로 비교: 두 곡선의 유사도를 시간 매개화와 무관하게 재는 표준 도구가 discrete Fréchet distance (Eiter & Mannila, 1994)와 DTW (Berndt & Clifford, 1994)입니다. "언제 지나갔는지"를 완전히 지우고 "어디를 지나갔는지"만 봅니다.
레이싱/주행 벤치마크 관행: 자율 레이싱 서베이(Betz et al., 2022)나 CARLA Leaderboard 방식 모두 공간 이탈(lateral deviation)과 시간 성과(lap time / route completion)를 별도 축으로 분리 보고하지, 하나로 뭉치지 않습니다.

2. 왜 CTE 평균조차 time이 이겼나 — 여기 숨은 편향 두 개
정성적으론 pos가 커브를 더 잘 타는데 CTE 평균은 time이 근소 우위(0.051 vs 0.059)인 이유:

프레임 평균의 시간 가중 편향: CTE를 프레임 평균하면 "느리게 지나간 구간"이 과대 가중됩니다. 직선이 전체의 대부분이라 커브의 corner-cutting은 평균에서 희석되고, 만약 pos가 커브에서 더 감속한다면 커브 프레임 수가 늘어 자기 오차가 더 많이 카운트되는 역차별까지 생깁니다. CTE는 시간이 아니라 호길이(arc-length)로 평균해야 합니다: ∫|cte| ds / S.
부호 소거: time mode의 선제 조향(코너 안쪽 파고들기)은 커브에서 체계적으로 한쪽 부호의 CTE를 만드는데, 절대값 평균은 이 편향(bias)을 무작위 흔들림과 구분 못 합니다.

3. 제안하는 공정 지표 세트 (보고서용)
Primary — 공간(매개화 불변), 이게 승패 판정:

호길이 평균 CTE ∫|cte|ds/S + max/p95 CTE (최악 이탈)
곡률 조건부 CTE: |κ| 구간별(직선/완커브/급커브) 분리 리포트 — corner-cutting은 고곡률 빈에서만 드러납니다. 사용자가 눈으로 본 차이가 정량화되는 곳이 여기일 확률이 높습니다.
커브 구간 부호 있는 CTE 평균: 0에서 벗어난 편향 = 체계적 안쪽/바깥쪽 치우침 = 선제 조향의 정량 증거
discrete Fréchet distance: 매개화-불변 최악 거리, 한 숫자 요약용

Secondary — 시간/속도, 별도 축으로만:

lag, 완주 시간비, v 프로파일 RMS — "time도 맞으면 best"의 채점은 여기서. primary와 합산 금지.

Tertiary — 안정성 (선언한 목표 "안정적 주행"의 지표):

조향 변화율 RMS, 횡가속/저크, 종료율(이탈/전복)

Stress set — 최종 목표("말이 안 되는 경로에서 합리적 속도 추론")의 직접 검증:

비합리 곡률 경로 세트에서 완주율 + max CTE + 종료율. CARLA식 composite(완주율 × 페널티)로 한 숫자화 가능.

### 1.2 문헌이 말하는 "좋은 Privileged Information"의 기준

1. **value에 인과적인 물리량**: 미래 reward(보상)를 좌우하지만 actor가 관측 못 하는 것 (contact force, friction, 미래 지형/경로).
2. **원시 센서값보다 가공된 dynamics state(동역학 상태)**: Lee 2020도 레이저 원시거리가 아닌 "terrain profile 샘플 + contact force"로 정제해 사용. 원시값 나열은 critic이 스스로 feature를 재발명해야 해서 비효율.
3. **차원이 크면 인코딩**: RMA는 env 인자를 latent 8D로 압축. 우리 33D는 encoder 없이 직결해도 되는 크기지만, 늘릴 거면 압축 고려.

### 1.3 우리 33D 재설계에의 적용

| 현행 | 문헌 대응 | 제안 |
|---|---|---|
| raycast distance×4 | 원시 센서값 — 문헌은 지형을 elevation(고도) 샘플/contact state로 정제 | **제거**. "바퀴 뜸"은 normal load(수직하중) N≈0이 대체 |
| wheel ω×4 (원시) | 문헌 표준은 **slip(슬립)** (접지 상태의 본질) | **slip ratio(슬립비) κ×4** (`physics.last_kappa`)로 교체 |
| suspension compression×4 | contact force (Lee 2020) | **normal load(수직하중) N×4** (`last_N`)로 교체 (물리 단위) |
| (없음) | 횡 미끄러짐 = cornering saturation(코너링 포화) 지표 | **slip angle(슬립각) α×4** (`last_alpha`) 추가 후보 |
| k_ex/a_ex ×20, lag ×1 | "미래 지형/경로 preview" 표준과 일치 | **유지** |

→ SDK(`genesis_vehicle` core.py)가 κ/α/N을 이미 계산하고 있어 **공짜 교체**. 이는 곧 실차 동역학에서 추정하는(TCS/ABS/ESC) 바로 그 상태들로, "practical한 privileged"라는 문제의식과 문헌 표준이 일치함.

### 1.4 [κ, α, N] 설계의 근거 (유사 논문)

**① 제1원리 — 타이어 모델의 충분 통계량(sufficient statistics)**: 표준 타이어 모델(Pacejka Magic Formula 계열, 우리 SDK 포함)에서 타이어 힘은 정확히 **F = f(slip ratio κ, slip angle α, normal load N)** 로 계산된다. 즉 [κ,α,N]은 차량 동역학의 힘 생성을 결정하는 **은닉 상태 그 자체** — critic에 이 셋을 주는 것은 "value를 좌우하지만 actor가 관측 못하는 물리량"이라는 privileged 원칙(§1.2-1)의 교과서적 적용이다.

**② 차량 도메인 직접 선례 — GT Sophy (Wurman et al., Nature 2022)**: GT Sophy의 차량 상태 입력이 정확히 이 조합이다 — *"three-dimensional velocity, angular velocity, acceleration, **the load on each tire, and tire-slip angles**"*. 나아가 reward의 tire-slip penalty도 **4바퀴 slip angle + slip ratio**로 계산. 즉 [바퀴별 하중 + 슬립] = Nature급 차량 RL이 검증한 car-state 설계다. (차이점: GT Sophy는 배포도 게임 내라 이를 **actor 관측**으로 썼고, 우리는 배포 시 관측 불가라 **critic privileged**로 배치 — 비대칭 AC 관례(Pinto 2017)에 따른 배치 차이일 뿐 정보 설계는 동일.)

**③ 한계 주행 RL 선례**: High-speed Autonomous Drifting RL (Cai et al., RA-L 2020, arXiv:2001.01377) — 한계 주행에서 **side slip angle을 핵심 state**로 사용 (최대 43° 슬립 제어). Reference-Free Formula Drift (arXiv:2410.20990, 2024) — slip 기반 **tire energy**를 보상으로 사용. 슬립이 한계 거동의 본질 변수라는 도메인 합의.

**④ locomotion 전이 관례**: Lee 2020(Science Robotics)의 privileged = **contact force**(우리 N의 보행 대응물) + friction — §1.1 표 참조. 접촉/하중을 privileged로 주는 것이 로봇 RL의 표준 레시피.

요약: [κ,α,N]은 (i) 타이어 물리의 충분 통계량이고, (ii) GT Sophy가 동일 조합을 car-state로 실증했으며, (iii) 한계주행 RL·locomotion privileged 관례와 모두 정합 — 반면 raycast 원시거리를 privileged로 쓴 차량 RL 선례는 찾지 못함.

### 1.5 실차에서 [κ, α, N]을 어떻게 재나 — sim2real 센서 계층

privileged 설계의 sim2real 타당성 점검: "시뮬에서만 아는 값"이 실차에서 **측정/추정 가능한 값인가**. 세 계층으로 나뉜다.

**계층 1 — 양산차(production) 장비만으로 (추정)**
| 상태 | 센서/방법 | 비고 |
|---|---|---|
| slip ratio κ | **WSS(Wheel Speed Sensor, ABS 톤휠)** 4개 + 차속 기준 → κ = (ωr − v)/v 직접 계산 | 전 양산차 표준. TCS/ABS가 실시간으로 쓰는 그 값 |
| slip angle α | 직접 측정 불가 → **IMU + WSS + steering angle의 observer(관측기) 추정** — EKF/sliding-mode 기반 sideslip estimation | ESC(차체자세제어)가 하는 방식. 문헌 다수 (MDPI 2025 interval observer, arXiv:2206.15119 model vs data-driven 비교) |
| normal load N | 직접 측정 불가 → **정적 배분 + IMU 가속도(a_x, a_y) 기반 load transfer(하중 이동) 추정** | 센서 0원 추정기. 서스 스트로크 센서 있으면 정밀도 ↑ |

**계층 2 — 계측 차량(instrumented test vehicle, R&D 표준)**
| 상태 | 장비 | 비고 |
|---|---|---|
| α (차체 sideslip) | **Kistler Correvit S-Motion** (비접촉 광학 지면속도 센서) — 종/횡 속도와 sideslip을 ±0.1° 정밀 직접 측정 | 차량동역학 시험의 사실상 표준 |
| α (대안) | **RTK-GNSS 듀얼안테나 + INS** (OxTS RT 계열) | heading vs 진행방향 차 → sideslip |
| κ, α, N, 타이어 힘 전부 | **Kistler RoaDyn WFT(Wheel Force Transducer)** — 휠 6분력 직접 측정 | 가장 완전하지만 고가·시험 전용 |
| N | **서스펜션 변위 센서(linear potentiometer)** → 스프링·댐퍼 힘 환산 | 모터스포츠 표준 |

**계층 3 — 신흥: Smart Tire(스마트 타이어)**
타이어 내장 가속도계/압전 센서로 접지면에서 **수직·종방향 접촉력과 슬립을 타이어가 직접 보고** (Pirelli Cyber Tyre 류). 문헌: *On the benefit of smart tyre technology on vehicle state estimation* (Vehicle System Dynamics 2022) — smart tire 신호가 sideslip/하중 추정 필터 성능을 유의미하게 개선. *Accelerometer Based Method for Tire Load and Slip Angle Estimation* (arXiv:1901.08049).

**아키텍처 함의 (이 계층 구조가 우리 설계를 정당화)**
1. **critic-privileged 배치가 정확히 맞는 이유**: 계층 1(양산)에서 α·N은 노이즈 있는 "추정"만 가능 → actor 관측으로 넣으면 sim(정확값)-real(추정값) 갭이 생김. critic에만 넣으면 **배포 시 아예 불필요** — 학습 때 시뮬 정확값의 이득만 취하고 sim2real 갭은 0.
2. **중간 경로 — RMA식 adaptation module**: 실차 배포에서 privileged 이득을 살리고 싶으면, Kumar 2021 패턴대로 **관측 가능한 이력(IMU+WSS 시계열) → privileged latent를 회귀하는 student 추정기**를 붙이는 2단계 증류가 표준. 계층 1 센서만으로 구현 가능.
3. **검증 경로**: 계층 2 장비(Correvit + WFT)는 real2sim 캘리브레이션 단계에서 시뮬 타이어 모델(κ,α,N→F)의 실측 대조에 쓰는 것이 정석 — privileged 값 자체의 real ground truth 확보 수단.

## Part 2 — "같은 환경 + MPPI/GA 배치" vs "멀티씬 PPO" (교수 피드백)

### 2.1 관련 논문

**Sampling-based(샘플링 기반) — MPPI 계열**
- **Williams et al. 2016-17**, *MPPI Control using Covariance Variable Importance Sampling* (arXiv:1509.01149) + *Information-Theoretic MPC* (ICRA 2017): MPPI 원조. **GPU에서 수천 rollout(롤아웃) 병렬 샘플링 → softmax 가중 평균 → 매 스텝 replanning(재계획).** 오프로드 공격 주행 실증. 단, 학습되는 policy(정책) 없음(training-free, 매번 재계획).
- **DIAL-MPC 2024** (arXiv:2409.15610, LeCAR Lab): diffusion식 annealing(어닐링)을 얹은 full-order sampling MPC. 사족보행 torque(토크) 제어에서 **RL policy보다 40% 우수** (training-free). "모델이 정확하고 온라인 계산이 허용되면 sampling MPC가 RL을 이긴다"는 최신 증거.

**Evolutionary(진화)/Genetic(유전) 계열 — parameter space(파라미터 공간) 탐색**
- **Salimans et al. 2017**, *Evolution Strategies as a Scalable Alternative to RL* (arXiv:1703.03864): OpenAI-ES. parameter perturbation(파라미터 섭동) ×population(개체군) → return(리턴) 순위 가중 업데이트. 수천 워커 선형 확장(통신은 스칼라만). **단 sample efficiency(샘플 효율)는 policy gradient(정책 경사) 대비 3~10배 열세.**
- **Such et al. 2017**, *Deep Neuroevolution* (Uber): 단순 GA도 심층망 RL에서 경쟁력. exploration(탐색)이 어려운(deceptive reward, 기만적 보상) 문제에서 강점.
- **하이브리드**: **ERL** (Khadka & Tumer 2018), **CEM-RL** (Pourchot & Sigaud, ICLR 2019, arXiv:1810.01222) — evolutionary(안정·탐색) + policy gradient(샘플 효율)를 population 안에서 결합.
- **GPU 가속 프레임워크**: evosax(JAX ES 30종), EvoRL (arXiv:2501.15129), EvoX — "GPU batch = population 평가"로 쓰는 최신 인프라. Brax에서 ES vs PPO 벤치마크 다수.

**Massively Parallel PPO(대규모 병렬 PPO) — 현행 방식의 근거**
- **Rudin et al. 2021**, *Learning to Walk in Minutes Using Massively Parallel Deep RL* (arXiv:2109.11978, CoRL): 단일 GPU 4096 env PPO로 ANYmal 보행 4~20분 학습. **Isaac Gym 생태계의 사실상 표준이 "수천 동질 env + PPO"**이며, 우리 512env 구조의 직계 조상.

### 2.2 교수 피드백에 대한 우리 상황 분석

**(a) "서로 다른 환경이라 batch가 비효율"이라는 전제부터 점검 — 우리 구현은 이미 homogeneous batch(동질 배치)임.**
멀티씬이라 해도 전 env가 **하나의 Genesis 씬, 하나의 p00 mesh, 동일 물리 파라미터**를 공유하고, env 간 차이는 reference CSV 텐서 인덱스(추종할 궤적)뿐이다. 물리 스텝·raycast·타이어 모델은 완전히 균일한 단일 batch 커널로 돈다. 즉 Rudin류 "수천 동질 env"와 GPU 활용 구조가 같고, **씬이 달라서 생기는 batch divergence(배치 발산)는 없다.** 교수님이 우려한 그림(씬마다 다른 시뮬 인스턴스가 각자 도는 것)과는 다른 구현이라는 점을 먼저 공유할 가치가 있음.

**(b) MPPI 방식은 이미 파이프라인 1단계에서 쓰고 있다 — 문제는 산출물의 종류.**
"같은 환경을 GPU batch로 MPPI"는 정확히 우리 **golden mining** 단계다 (per-scene 수천 rollout → 최적 입력 시퀀스). MPPI/GA가 만드는 것은 **그 씬의 궤적/계획**이고, PPO가 만드는 것은 **씬을 일반화하는 closed-loop(폐루프) policy**다:

| | MPPI (동일씬 배치) | 멀티씬 PPO (현행) |
|---|---|---|
| 산출물 | 씬별 최적 입력 시퀀스 | 단일 generalist policy (26씬 + 미학습씬) |
| 미학습 씬 | 그 씬에서 재최적화 필요 (수천 rollout) | zero-shot(제로샷) (p117 실증: time lagfix 완주) |
| 실시간 배포 | 매 스텝 수백~수천 rollout 온라인 | MLP forward 1회 |
| closed-loop 복구 | replanning으로 해결 (모델 정확할 때) | policy가 자기실수 분포에서 학습 |

LbC·Roach가 보여준 표준 레시피가 "privileged expert(계획기) → 학생 policy distillation"인데, **우리는 이미 MPPI(expert) → BC(distillation) → RL(closed-loop 보정)로 그 레시피의 확장판을 구현한 상태**다. MPPI로 회귀하면 3단계가 1단계로 퇴화한다.

**(c) GA/ES의 실질 검토 — 우리 actor가 작아서(≈2만 파라미터) 적용 자체는 현실적.**
- 성립하는 이유: ES는 population=env 수로 잡으면 512env가 그대로 512 perturbation 평가기가 됨. backpropagation(역전파) 불필요, 코드 단순.
- 불리한 이유 ①: **우리 병목은 물리 스텝이지 backward가 아니다** (minibatch 업데이트는 rollout 대비 저렴). ES의 최대 장점("업데이트가 싸다")이 우리 프로파일에선 이득이 안 됨.
- 불리한 이유 ②: PPO는 수집한 데이터를 5 epoch 재사용(importance ratio clipping, 중요도비 클리핑)하지만 ES는 perturbation당 **풀 에피소드 평가**가 필요 → Salimans 자체 보고로 sample efficiency 3~10배 열세. 같은 GPU 시간에서 도달 성능이 낮을 공산.
- 유리할 수 있는 경우: reward가 deceptive하거나(local optima, 국소 최적) gradient가 불안정할 때(우리 lagfix 前 exploration collapse 같은 상황). ES는 rank-based(순위 기반)라 reward 스케일에 둔감하고 collapse가 없음.

### 2.3 제언 (교수 피드백을 살리는 방법)

1. **주장 정정 공유**: "우리 멀티env는 이미 단일 씬·homogeneous 물리 batch이며 GPU 활용 구조는 Isaac Gym 표준(Rudin 2021)과 동일" — batch 효율 우려는 구현상 이미 해소됨. GPU utilization(활용률)을 더 올릴 레버는 알고리즘 교체가 아니라 **env 수 스케일업**(512→2048+)이다.
2. **ES ablation(절제 실험) (1일, 선택)**: 동일 rollout 인프라에서 actor만 OpenAI-ES(population 512)로 학습해 PPO lagfix와 fm/cte/lag 비교. 교수 피드백에 대한 정량 답변이 되고, 코드 변경이 작다 (업데이트 루프만 교체).
3. **MPPI의 올바른 재활용처**: policy 대체가 아니라 — (i) 미학습 씬의 golden 재마이닝(라벨 생산), (ii) DAgger식 corrective label(교정 라벨) 생성기, (iii) 시뮬 내 oracle(상한) 성능 측정기. DIAL-MPC가 보여주듯 "모델=시뮬레이터 그 자체"인 우리 환경에서 MPPI는 사실상 성능 상한이므로, **policy 평가의 baseline(기준선)**으로 쓰면 보고서 설득력이 올라간다.

## Part 3 — 평가 지표: Spike 정량화(Q3) & Time/Position 공평 비교(Q4)

### 3.1 Spike(국소 최악 오차) 정량화 — 가능하며, 표준 기법이 이미 있음 (Q3)

path tracking(경로 추종) 평가 문헌은 mean(평균)과 별개로 worst-case(최악) 지표를 표준으로 병기한다:

| 지표 | 정의 | 근거 문헌 |
|---|---|---|
| **MLE(Maximum Lateral Error, 최대 횡오차)** | 전 구간 max CTE | *Review and performance evaluation of path tracking controllers* (2021) — MLE/MOE/ALE/AOE가 표준 4종 |
| **percentile(백분위) CTE** | p90/p95/p99 | subarctic forest navigation (arXiv:2111.13981): **곡률 구간별 p90 CTE** 보고 (저곡률 <0.6m vs 중곡률 1.0m) — 정확히 "코너에서 얼마나 벌어지나"의 문헌 선례 |
| **CVaR@α (Conditional Value at Risk, 조건부 기대 손실)** | 최악 α% 프레임의 평균 오차 | risk-sensitive RL 표준 (arXiv:2206.04436 등). max보다 outlier(이상치) 1개에 강건 |
| **segment(구간) 단위 peak** | 코너 세그먼트별 최대 오차 | racing 문헌 관례: "max CTE 1.3m **around bends** vs mean 0.323m"처럼 발생 위치와 함께 보고 |

**우리 적용 (즉시 구현 가능)**: `ref['k']`(곡률)가 이미 있으므로 —
1. **curvature-gated segmentation(곡률 게이트 구간화)**: |k| > k_thr 연속 구간 = 코너 세그먼트로 자동 분할.
2. 세그먼트별 **peak CTE / peak lag**와 **spike 상승폭**(peak − 코너 진입 시점 오차) 산출.
3. 요약: worst-corner 값 + CVaR@10% + p95. → "p134 좌코너에서 lag가 최대 1.4m까지 벌어짐"이 재현 가능한 수치로 표준화됨.

### 3.2 Time/Position 공평 비교 — 문헌 (Q4)

**문제의 구조**: 우리 fm(frame-matched error)은 "같은 시각 인덱스"끼리 비교 → time mode의 목적함수와 정렬돼 있고, position mode에는 schedule(스케줄) 편차가 기하 오차에 합산되는 unfair(불공평)한 지표다 (fm² ≈ cte² + lag²).

같은 문제를 다룬 문헌 계보:

| 문헌 | 기여 | 우리와의 대응 |
|---|---|---|
| **Sturm et al. 2012**, *A Benchmark for the Evaluation of RGB-D SLAM Systems* (IROS) | **ATE(Absolute Trajectory Error)**(시간 매칭 전역 오차)와 **RPE(Relative Pose Error)**(국소 상대 오차)를 **분리 보고**하는 관례 확립 | ATE=우리 fm과 동일 사상. "하나의 지표로 합치지 말고 성격이 다른 오차를 분리 보고하라"의 원조 |
| **DTW(Dynamic Time Warping, 동적 시간 신축)** (Toohey & Duckham 2015 survey) | 시간축 신축을 허용하는 최적 매칭 후 거리 → **시간 지연에 불변인 기하 유사도** | position mode의 "위상을 희생한 경로 충실도"를 정당하게 평가 |
| **Discrete Fréchet Distance(이산 프레셰 거리)** | 순서 보존 min-max 기하 거리 | 시간 프리 + worst-case → Q3(spike)와도 연결 |
| **Hausdorff Distance(하우스도르프 거리)** | 순서 무시 순수 형상 거리 | 참고용 하한 (순서 무시라 루프 궤적에 취약) |
| *A comparative analysis of trajectory similarity measures* (2021) | 각 measure를 **relative vs absolute time/space 처리** 기준으로 분류 | 이 분류축이 정확히 우리 time/position 축 — "어떤 지표든 시간 처리 방식에 대한 입장을 내포한다"는 것이 문헌의 결론 |

### 3.3 내 제안 (Q4에 대한 우리 상황 방법론)

**대전제: 중립적인 단일 스칼라는 존재하지 않는다.** fm은 time에, DTW/Fréchet은 position에 유리하다. 문헌(Sturm의 ATE/RPE 분리)이 시사하는 유일하게 공평한 방법은 **직교 2축 분해 보고**다:

- **기하 축(geometry)**: 경로에서 얼마나 벗어났나 — CTE(투영 수직거리) mean/p95/max
- **스케줄 축(schedule)**: 시간표에서 얼마나 어긋났나 — lag mean/max, 완주시간 비율

fm은 이 둘의 합성이므로 headline(대표 지표)이 아니라 참고치로 강등.

**제안 ① Arc-length Matched Error(호길이 매칭 오차)** — fm의 position-fair 쌍둥이.
차량 진행 호길이 s_car(t)와 동일한 호길이의 ref 지점을 매칭해 기하 오차 계산(시간 인덱스 대신 호길이 인덱스). 스케줄 성분이 완전히 제거된 fm 대응물. 별도로 **completion time ratio(완주시간 비율)** = T_car/T_ref 를 스케줄 스칼라로 병기. 구현: `arc_s` 이미 있음 → 수십 줄.

**제안 ② DTW 한 번으로 2축 동시 산출.**
DTW alignment(정렬) 후 — (i) 정렬 경로상 평균 거리 = 기하 점수, (ii) warping path의 대각선 이탈량 mean|i−j|·(호길이/프레임) = 스케줄 점수. 한 계산으로 두 축이 나오고, 두 mode 모두 자기 강점 축에서 점수를 받는다. O(N²)이지만 N≈400~700이라 순간.

**제안 ③ Q3 spike 지표를 공평 비교에 포함.**
코너 세그먼트별 peak CTE(기하 spike)와 peak lag(스케줄 spike)를 분리 보고 — "position은 코너 기하를 지키고, time은 코너 스케줄을 지킨다"는 트레이드오프가 수치로 드러남.

**제안 ④ 보고 형식**: 최종 표를 [CTE mean | CTE p95 | corner peak CTE | lag mean | corner peak lag | 완주시간비 | brake usage]로 구성 — 단일 승자 선언 대신 2축 프로파일 비교. 현행 `0_eval`이 이미 cte/lag를 따로 계산하므로 추가 구현은 percentile/segment 통계 + DTW 함수 정도.

## Part 4 — "1:1 시간 대응은 필수가 아니다": Trajectory Tracking vs Path Following, 그리고 RL (Q5)

### 4.1 이론적 근거 — 제어 문헌의 고전적 구분

| 개념 | 정의 | 우리 대응물 |
|---|---|---|
| **Trajectory Tracking(궤적 추종)** | 시간 파라미터화된 reference — **매 시각 t에 있어야 할 지점**이 지정됨 (1:1 대응) | 우리 **time mode** (i_sched 위상 시계) |
| **Path Following(경로 추종)** | 기하 경로만 지정, **timing law(시간 배분)는 제어기의 자유** | 우리 position mode의 극한 (schedule 완전 제거) |

- **Aguiar, Hespanha & Kokotović 2005**, *Path-following for nonminimum phase systems removes performance limitations* (IEEE TAC): **핵심 정리 — trajectory tracking에는 unstable zero dynamics(불안정 영점 동역학)가 부과하는 L2 추종오차 하한이 존재하지만, 같은 시스템이라도 path following으로 문제를 바꾸면 이 한계가 사라진다.** 즉 "1:1 시간 대응을 포기하면 이론적으로 더 잘 따라갈 수 있다"는 것이 정리 수준에서 증명돼 있음. 차량(후륜조향 관점의 non-minimum phase 성질)에 직접 관련.
- **Faulwasser & Findeisen**, *NMPC for Constrained Output Path Following* (arXiv:1502.02468): MPC 관점의 동일 결론.

### 4.2 실무적 구현 — Progress 기반 제어/RL

| 논문 | 방식 | 시사점 |
|---|---|---|
| **Liniger et al. 2015**, *MPCC(Model Predictive Contouring Control) for Autonomous Racing* (ETH) | 경로 진행도 θ를 **가상 상태로 증강**, contouring error(횡)·lag error(진행 결합)를 벌점하며 **progress(진행)를 최대화** | "출력 길이 = 입력 프레임 수" 대응을 버리고 진행도를 제어기가 스스로 결정하는 표준 설계. 우리 lag 보상은 이것의 schedule 고정판 |
| **Wurman et al. 2022**, *GT Sophy* (Nature 602) | progress 보상 기반 RL — 시간 인덱스 궤적 추종 없이 **전 구간 인간 챔피언 초과** | reference 없는 progress RL의 최상급 실증 |

### 4.3 이 관점의 정량 Metric (Q5 후반)

timing 자유를 준 순간 fm·lag는 **정의상 무의미**해지고(스케줄이 없으므로), racing/path-following 문헌의 표준 지표로 교체됨:

| Metric | 정의 | 문헌 출처 | 우리 구현 난이도 |
|---|---|---|---|
| **Completion Time(완주 시간)** / lap time | 경로 끝(s_car ≥ 총 호길이) 도달까지 시간. 비교는 ratio(T_policy/T_ref 또는 T_oracle) | racing 표준 (GT Sophy·Swift의 headline) | traj에서 사후 계산 가능 — `_eval_fair`에 1줄 추가 |
| **Contouring Error(윤곽 오차)** mean/p95/max | 경로까지 수직거리 — 우리 **cte와 동일** | MPCC | ✅ 이미 구현 |
| **Progress Rate(진행률)** | 단위시간당 경로 호길이 진행 (m/s) | MPCC/racing | s_car 미분 — 사후 계산 |
| **Success/DNF율, 경계 위반율** | 완주 실패·트랙 이탈(cte > 임계) 비율 | racing 표준 | cte 임계 카운트 |
| **Control Effort/Smoothness(제어 노력/부드러움)** | mean\|ΔS\|, mean\|Δu\| (steering effort) | path tracking 리뷰(SE/MSE) | act 로그에서 계산 |
| **Speed Envelope 위반** | v가 곡률 한계속도 √(a_lat_max/\|k\|) 초과한 비율 — timing 자유 시대의 "과속" 감시 | 차량동역학 관행 | ref k로 계산 |
| **vs Oracle gap** | MPPI(시뮬 상한, Part 2.3)와의 완주시간/오차 격차 % | DIAL-MPC·racing 관행 | golden 대비 비교 |

핵심: **Part 3의 2축(기하/스케줄) 중 스케줄 축이 "lag(스케줄 준수)"에서 "completion time(절대 시간 성능)"으로 교체**되는 것. 기하 축(cte 계열)은 그대로 유효하다.

### 4.4 우리 상황 적용 — 단, 목적 정의가 먼저

이 관점을 채택하기 전에 **task 사양의 분기점**을 명시해야 함:

- **(가) real2sim 재현이 목적** (blender 궤적을 **속도 프로파일까지** 재현): 1:1 시간 대응이 목적 그 자체 — path following으로 바꾸면 과제를 바꾸는 것. 현행 time mode가 정답.
- **(나) "그 경로를 잘/빠르게 주행"이 목적**: reference는 기하로 강등되고, timing은 policy 자유 → progress 보상 RL(GT Sophy식)이 문헌상 우월. 이때 v_target·lag 보상 제거, progress + contouring + envelope 보상으로 교체. **Song 2023이 보여주듯 이 경우 RL이 궤적추종 방식을 이기는 것이 일반적.**
- **(중간) 속도 프로파일은 소프트 가이드**: MPCC처럼 progress 최대화 대신 v_target을 soft 참조로 두고 lag 하한만 제거 — 우리 position mode + lag 보상 완화가 이 중간지대. 미리돌기·저속씬 강건성 문제의 근본 해법 후보이기도 함.

우리 파이프라인은 (가)로 설계돼 왔음(golden mining의 fm 목적함수부터). 교수 피드백의 "RL이 더 나은 대안"은 (나)로의 목적 전환 제안으로 읽는 것이 정확하며, 이는 알고리즘 교체가 아니라 **제품 정의 결정**이다. (나)로 가면 Part 3에서 만든 지표 체계 중 스케줄 축만 completion time으로 갈아끼우면 평가 인프라는 그대로 재사용된다.

### 4.5 결정 및 실측 (2026-07-15)

**결정: RL 관점의 평가 metric = (나) path-following 표준 채택** (구현: `0_eval/_eval_pathfollow.py` — 에피소드 1.5N 연장 + 자기교차 안전 단조 matcher). BC/재현 서사는 (가) 유지.

26씬 실측 (time lagfix it500 vs pos gate2 it600):

| | comp(T/T_ref) | DNF | cte | p95 | max | 경계위반 | steer(mrad/f) |
|---|---|---|---|---|---|---|---|
| time | 0.972 | 0 | 0.044 | 0.134 | 0.85 | 0.6% | 18.45 |
| **pos** | 0.978 | 0 | **0.039** | **0.113** | **0.29** | **0.0%** | **12.27** (26/26승) |

- **표준을 바꾸면 순위가 뒤집힌다**: 스케줄 강제를 풀면 pos가 기하·smoothness 우세, 완주시간 동급(둘 다 ref보다 ~3% 빠름). time의 미리돌기 corner-cut이 이 관점에선 worst-case 기하 오차(max 0.85)로 노출.
- **아티팩트 정정**: 시간고정 평가에서 pos의 기하 악화(p120 cte_p95 3.24)는 **past-end 아티팩트**(조기 완주 후 잔여 프레임이 경로 끝 너머 주행) — 완주시점 컷 시 0.28. 시간고정 지표로 mode 비교 시 반드시 완주시점 컷 필요.
- **단서**: pos는 미학습 씬 정지 데드락(DNF) 미해결 — 이 표준에서도 DNF는 실격이므로, pos 채택엔 강건성(스폰노이즈 등 학습측 해결)이 선결 조건.

## 5. 종합 인사이트

**Q1 (Privileged Information)**: 문헌 합의는 명확하다 — privileged는 "**actor가 못 보지만 value에 인과적인 물리량**"이고, locomotion 계열 표준은 contact force·slip·지형/경로 preview다. 원시 raycast 거리를 그대로 넣는 사례는 드물며(정제해서 넣음), 사용자의 "over-feature" 직관은 문헌과 일치한다. SDK가 이미 κ/α/N을 계산하므로 [dist,ω,comp]→[κ,α,N] 교체는 비용이 거의 0인 문헌 정합 리팩터다. 단, privileged input은 critic 전용이라 개선 효과는 "value estimation variance(분산) 감소"에 한정 — lagfix급 도약은 기대하지 말고 **spawn noise(스폰 노이즈) 재설계와 묶어 한 번의 재학습**으로 처리하는 것이 경제적.

**Q2 (MPPI/GA vs PPO)**: 이 질문의 본질은 알고리즘 취향이 아니라 **"무엇을 만들 것인가"**다. 산출물이 씬별 최적 궤적이면 MPPI가 맞고(이미 마이닝에서 사용 중), 미학습 씬까지 일반화하는 실시간 policy면 policy learning이 맞다. 우리 파이프라인은 두 방식을 이미 역할 분담시켜 놓았다(MPPI=라벨 생산, PPO=closed-loop 일반화). GPU batch 효율 관점에서도 물리 스텝이 병목인 이상 MPPI/GA/PPO 어느 쪽이든 같은 비용을 지불하며, 우리 구현은 이미 homogeneous batch다. 교수 피드백에서 취할 실익은 (i) ES ablation으로 정량 비교 카드를 만들고, (ii) MPPI를 oracle baseline으로 승격시켜 "RL policy가 상한 대비 몇 %인가"를 보고서에 넣는 것이다.

**Q3·Q4 (평가 지표)**: spike는 curvature-gated segment(곡률 게이트 구간) 단위 peak + CVaR@10%로 즉시 정량화 가능하고(문헌 선례: 곡률 구간별 percentile CTE), time/pos 공평 비교의 정답은 "더 공평한 단일 지표 찾기"가 아니라 **기하/스케줄 2축 분해 보고**(Sturm ATE/RPE 분리 관례)다. fm은 time에 유리한 합성 지표임을 명시하고 headline에서 내리는 것이 지적으로 정직한 보고 방식이며, arc-length matched error + 완주시간비(제안 ①) 또는 DTW 2축(제안 ②)이 position mode의 실제 강점(경로 충실도)을 처음으로 공정하게 드러낼 것이다.

**Q5 (1:1 대응 불필요론)**: 이는 제어 이론의 고전적 구분(trajectory tracking vs path following)이며, **timing 자유가 이론적 성능 한계를 제거한다는 것은 정리로 증명돼 있고**(Aguiar 2005), progress 보상 RL이 시간 궤적 추종을 이긴다는 실증도 최상급 저널에 있다(Song 2023 Science Robotics, GT Sophy, Swift). 단 이 관점의 채택은 알고리즘 선택이 아니라 **목적 정의의 전환**이다 — real2sim "재현"(속도 프로파일 포함)이 목적이면 1:1 시간 대응이 과제 그 자체이고, "그 경로를 잘 주행"이 목적이면 progress RL이 문헌상 정답이다. 평가도 그에 맞춰 스케줄 축을 lag→completion time으로 교체하면 되고, 기하 축(cte 계열)과 우리 평가 인프라는 그대로 재사용된다. 우리 time/position 두 mode는 사실 이 스펙트럼의 양 끝을 이미 부분적으로 커버하고 있다.
